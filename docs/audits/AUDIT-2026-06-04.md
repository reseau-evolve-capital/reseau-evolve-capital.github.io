## 3. Nouveaux défauts & régressions

> **BUG** = bug objectif · **REG** = régression introduite par la passe de corrections. Preuve = capture + sonde.

### 🟠 B1 — `/admin/cotisations` : clés React dupliquées → timeline club corrompue · LOG/ROB/A11Y/DG · BUG · MAJEUR

- **Symptômes runtime** : au chargement par défaut de `/admin/cotisations`, **12 erreurs console** : `Encountered two children with the same key: 2025-1 … 2025-6`. Visuellement, l'année **2025 affiche 23 cellules au lieu de 12** : labels « J J J F F F M M M A A A M M M J J J A S O N D » — janvier→juin **répétés ×3** (un par membre du club), juillet→décembre une fois. Les autres années (2024/2023/2022…) affichent une ligne propre de 12 mois (`reaudit-admin-cotisations-1440.jpeg` ; Next DevTools « 6 Issues »).
- **Preuve que ce n'est PAS une donnée corrompue** : la DB contient **exactement 1 ligne par mois** pour 2025 (`SELECT … GROUP BY year,month` → 12 lignes, n=1 chacune). Le défaut est purement au rendu.
- **Cause racine** :
  - `apps/web/app/(app)/admin/cotisations/page.tsx:22` appelle `getClubContributionsTimeline(supabase, ctx.clubId, null)` — `membershipId = null` → **tous les membres du club**.
  - `apps/web/lib/data/admin.ts:227-261` (`getClubContributionsTimeline`) verse les ~90 `contribution_months` (3 membres × ~30 mois) dans `buildTimelineYears(months)` sans agrégation : un groupe-année contient donc plusieurs entrées au même `month`.
  - `packages/ui/src/organisms/ContributionsTimeline/ContributionsTimeline.tsx:134` clé chaque cellule par ``key={`${group.year}-${m.month}`}`` → collisions (`2025-1` ×3). React avertit (« children may be duplicated and/or omitted ») et **duplique/omet** des nœuds, d'où les 23 cellules.
  - L'état cassé persiste même après filtrage par membre (les clés dupliquées brisent l'identité de réconciliation ; `useAdminContributions` sème `initialData` = payload « tous membres » corrompu). Vérifié : load frais filtré sur 1 membre → toujours 23 cellules + erreurs `2025-1…2025-6`.
- **Verdict design-lead** : **corriger** — bug objectif. La timeline `ContributionsTimeline` (per-membre, 1 mois = 1 cellule) est réutilisée pour des données multi-membres sans agrégation ni clé unique. Deux options : (a) agréger côté `getClubContributionsTimeline` (1 cellule/mois reflétant le statut club, dédupliquée) ; (b) rendre la clé unique ET adapter l'UX multi-membres. **Note** : la mémoire Sprint 6 décrit « admin cotisations = timeline club, filtre membre, stats » — l'intention est une timeline club, donc l'agrégation (a) est la voie cohérente. Tant que B1 n'est pas corrigé, la vue trésorier des cotisations est trompeuse (mois fantômes) et bruyante (storm console).

### 🟡 R1 — `/onboarding/tour` : débordement horizontal à 375px · RWD · REG · MINEUR

- **Preuve sonde (375/360px)** : `document.scrollWidth = 422 > clientWidth = 360` (overflow ~62px). Le wrapper `div.flex.min-h-screen` mesure 484px, la `section.max-w-[640px]` 452px ; comme le parent est `justify-center`, le contenu déborde symétriquement et **pousse le ProgressHeader « Étape 3 sur 3 » hors-champ à gauche** (`left:-21px`) (`reaudit-onboarding-tour-375.jpeg`, label tronqué « PE 3 SUR 3 »).
- **Cause racine** : `packages/ui/src/organisms/CarouselSlider/CarouselSlider.tsx:53-56` — la piste `div.flex.overflow-x-auto` n'a **ni `min-w-0` ni `w-full`** ; ses slides (l.58-63) sont `w-full shrink-0`. Un item flex avec `min-width:auto` se dimensionne sur le min-content (3 slides pleine largeur côte-à-côte), gonflant la `section` au-delà du viewport. `/onboarding/step-1` (sans carrousel) **ne déborde pas** (`scrollWidth=360`, section 224px) ; desktop 1440 **ne déborde pas** (section cappée à 640px < viewport). Donc spécifique **tour + ≤640px**.
- **Verdict design-lead** : **corriger** (1 ligne : `min-w-0` ou `w-full` sur la piste l.55). Impact : ProgressHeader F9 clippé sur mobile + scroll horizontal parasite. Mineur (l'écran reste fonctionnel, carrousel scrollable), mais c'est une régression liée à l'intégration tour+ProgressHeader.

### 🟡 N1 — `aria-valuenow` non arrondi sur la barre de quote-part (admin/members) · A11Y · BUG · MINEUR

- **Preuve sonde** : `/admin/members`, la `progressbar` « Quote-part de BAMBA Inès » porte `aria-valuenow="3.5000000000000004"` (flottant IEEE non arrondi), sans `aria-valuetext`. Le texte visible est correct (« 3,50 % »), mais un lecteur d'écran annoncerait « 3,5000000000000004 pour cent ».
- **Cause** : `packages/ui/src/atoms/ProgressBar/ProgressBar.tsx:20` `aria-valuenow={clamped}` propage la valeur brute (`clamped` = min/max sans arrondi).
- **Verdict** : **corriger** (arrondir `Math.round(clamped*10)/10`, ou fournir `aria-valuetext` formaté). Cosmétique a11y, non bloquant.

### 🟡 N2 — Nom accessible du bouton hero verbeux (dashboard) · A11Y · DIV · MINEUR

- **Preuve** : le `button` hero a pour nom accessible « Ta quote-part 12 345,67 € **Ta quote-part : 12345.67 euros** Mis à jour il y a 19 min » — le span `sr-only` `aria-live` (utile pour l'annonce live) est aussi concaténé dans le nom du bouton, le rendant long et redondant (et expose la valeur brute non formatée `12345.67`).
- **Verdict design-lead** : **différer/corriger léger** — soit sortir le span live du sous-arbre du bouton, soit donner au bouton un `aria-label` synthétique court. Faible enjeu, déjà noté en F4 de l'audit initial.
